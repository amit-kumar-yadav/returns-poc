Executive Summary

RLP (Reverse Logistics Platform) enables consumers to schedule doorstep returns and exchanges across multiple merchants in a single pickup session — one Dasher, one service fee, one QR code. Unlike existing solutions (Happy Returns, in-store drop-offs), the Dasher is pre-assigned to specific items before arrival, creating a structured handover that shifts liability precisely at scan time. A variable service fee — charged at commit, separate from item refunds — aligns incentives: consumers pay for the pickup service regardless of merchant inspection outcome, and merchants receive only items they've agreed to accept through their own return policies.

Problem Statement

Returns Are Broken for Everyone

E-commerce returns are expensive, friction-heavy, and poorly instrumented. Consumers haul boxes to UPS stores; merchants lose visibility after handoff; platforms generate no revenue from the reverse flow.

● Consumer Pain

Returning items from multiple merchants requires multiple trips, multiple labels, multiple windows. Consumers abandon eligible returns because the friction cost exceeds the refund value.

~30% of online purchases result in a return. Most are never completed.

● Merchant Pain

Unstructured returns create fraud exposure. Items arrive without pre-verification. Inspections are manual and asynchronous. Return policy enforcement is inconsistent downstream.

Return fraud costs merchants $100B+ annually in the US. Policy enforcement at inspection is the only defense.

● Platform Opportunity

The delivery network is already deployed. Dashers make millions of dropoffs daily. The reverse flow — pickup from consumer, delivery to merchant — requires only incremental infrastructure and creates a new recurring revenue stream.

Service fee per return = incremental platform revenue with no consumer subsidy.

Competitive Differentiation

Why RLP Is Different from Happy Returns

Happy Returns uses drop-off hubs and batch aggregation. RLP uses a pre-assigned Dasher network for doorstep pickup. The structural difference changes the entire liability and state machine design.

QR code role Discovery mechanism — staff looks up the return Handover confirmation artifact — backend resolves FxOrder from (DasherID + QR)

🎯
Core Insight: QR = Confirmation, Not Discovery

In Happy Returns, the QR tells staff what the return is. In RLP, the Dasher already knows their items — the QR scan is the moment liability shifts from consumer to Dasher. This inversion is the architectural foundation for how ReturnFxOrder and ConsumerReturn are decoupled.

💡
Why the Service Fee Is Non-Refundable by Design

The service fee is payment for the logistics service — the Dasher's time, the routing, and the platform infrastructure. It is not an escrow for the item value. Charging it at commit (before pickup, before inspection) aligns incentives: consumers who initiate returns complete them; Dashers are paid regardless of merchant decisions; merchants aren't penalized for legitimate rejections. This is the same model as a delivery fee being non-refundable if the restaurant closes after order placement.

Service Fee Model

Variable Fee Formula

The service fee is transparent, variable, and shown separately from item refunds before the consumer commits. It covers platform logistics cost and aligns incentives across all parties.

ServiceFee = f(items, merchants, exchanges)
$ 1.99 Base fee (per return session)
+ 0.50 × item count
+ 0.75 × (distinct merchants − 1)  [0 if single merchant]
+ 2.99 × exchange item count
= ServiceFee Charged at COMMIT · Non-refundable · Separate from item refunds
Example Calculation — Mixed Return Session
2 items from Merchant A, 1 item from Merchant B (3 items, 2 merchants)
Base fee $1.99
Per item (3 × $0.50) $1.50
Extra merchant (1 × $0.75) $0.75
Exchange items (0) $0.00
Total service fee (consumer pays at commit) $4.24
✓
Item refunds are always full historicalValue

Accepted items are refunded the full purchase price from the original CxOrder.historicalValue. The service fee is not deducted from refunds. This is displayed clearly in ReturnQuote so consumers see both numbers before committing.

QR Code Architecture

QR as Handover Confirmation, Not Discovery

The QR code design encodes a critical architectural decision: liability shifts at scan time, not at assignment time.

What the QR Code Contains
// ConsumerReturn.qrToken structure
QRPayload = {
  token: "cr_tok_A7x9...Pq2", // opaque
  consumerReturnId: "cr_...",
  issuedAt: 1721000000,
  expiresAt: 1721086400 // 24h window
}

// Resolution: backend lookup
resolve(dasherId, qrToken) → ReturnFxOrder
// → validate Dasher owns this FxOrder
// → transition items to IN_TRANSIT
Why This Design?
  • 1. One QR, multiple Dashers. If routing splits items across two FxOrders, each Dasher scans the same QR. The backend resolves each to their specific FxOrder — correct items, correct liability shift, no confusion.
  • 2. Scan = liability transfer. Before scan: consumer holds liability. After scan: Dasher's FxOrder holds liability. This maps directly to the physical handover — items change hands, state changes simultaneously.
  • 3. Consumer UX simplicity. Consumer sees one QR in Return History. They don't need to know about FxOrders, routing, or capacity. The platform manages complexity behind the token.
  • 4. Replay attack prevention. The resolve operation validates that (dasherId, qrToken) → a VALID, ASSIGNED FxOrder in the correct state. Stale or re-used tokens fail validation before any state transition.
Roadmap

Hub Inspection: Built Into the Model Today

The ReturnFxOrder.deliveryDestination: 'merchant' | 'hub' field is the only schema change needed to enable centralized hub inspection. The rest of the state machine is identical.

MVP — Now
Merchant-Direct Inspection
Dashers deliver returns to the originating merchant. Merchant staff inspect items manually. deliveryDestination: 'merchant'. InspectionResult created by merchant portal. This is the production architecture.
Phase 2 — Hub Pilot
Centralized Hub Receipt
Dashers deliver to a platform-operated hub instead of merchant. Hub receives items from multiple merchants in batch. deliveryDestination: 'hub'. All downstream state machines (ItemReturn, MoneyMovement, InspectionResult) are unchanged.
Phase 3 — Standardized Grading
Hub Inspection with Grading & Robotics
Hub implements standardized condition grading, photo documentation, and (eventually) robotic sorting. InspectionResult gains a grade field. Platform can offer merchant-configurable accept thresholds based on grade rather than binary accept/reject.
Phase 4 — Recommerce
Platform-Managed Resale
Graded items in acceptable condition can be relisted on the platform marketplace rather than returned to original merchant. New MoneyMovementEvent type: RESALE_CREDIT. Merchant earns partial value from resale instead of full refund obligation.
🏗
Zero schema migration for Phase 2

The ReturnFxOrder.deliveryDestination enum field is already in the MVP schema. Routing logic changes (dispatch assigns to hub address instead of merchant address), but no entity model changes are required to ship Phase 2. This was a deliberate architecture decision.

Architecture Decisions

Key Design Choices

Each decision below was made with explicit trade-offs in mind. The rationale is documented here for engineering leadership review.

1
ConsumerReturn ≠ ReturnOrder
Consumer intent layer decoupled from operational execution layer

ConsumerReturn owns the consumer-facing state: what items the consumer selected, their QR code, the service fee they paid, and the refund state they see in Return History. It is the "receipt" of the return transaction from the consumer's perspective.

ReturnOrder (and its ReturnFxOrder children) owns the operational execution: which Dasher is assigned, where items are routed, and what the physical delivery status is. It is the "fulfillment" layer.

These two can evolve independently. If routing needs to re-assign a Dasher, split items across two FxOrders, or re-route to a hub — none of that touches ConsumerReturn state. The consumer never sees the fulfillment complexity.

Consequence
Support tooling, consumer-facing status, and refund state are always consistent and readable from ConsumerReturn — regardless of operational re-routing events on the ReturnOrder side.
2
Service Fee Charged at Commit, Not at Pickup
Fee is for the pickup service, not the return outcome

The service fee is non-refundable and is charged the moment a consumer commits to a return — before any Dasher arrives, before any inspection occurs. This is a deliberate incentive alignment decision.

Fraud reduction: Consumers who pay upfront complete their returns at a significantly higher rate. Opportunistic "I'll see if they even come" behavior is eliminated.

Dasher earnings certainty: Dashers are compensated for their time regardless of whether the merchant accepts the items. This is analogous to a delivery fee being non-refundable if the recipient refuses the package.

Merchant protection: Merchants don't bear the cost of logistics for items they ultimately reject. The service fee sits entirely with the consumer-platform relationship.

Consequence
SERVICE_FEE MoneyMovementEvent is excluded from all reversal logic. Explicit guard in C&R service: "never reverse SERVICE_FEE type events." This must be enforced at the service layer, not just by convention.
3
ReturnFxOrder = Dasher Task, Not Merchant Partition
Assignment by routing/capacity, not by merchant boundary

A ReturnFxOrder represents one Dasher's pickup task. It is not one task per merchant. Items from multiple merchants can be in one FxOrder; one merchant's items can span multiple FxOrders across different Dashers.

This matches how forward delivery already works — a Dasher picks up from multiple restaurants in one batched trip. The routing engine optimizes for geography, capacity, and timing — not for merchant boundaries.

The implication: ReturnFxOrder carries a deliveryDestination that is the merchant (or hub) where items will be dropped. Items destined for different merchants will be in different FxOrders — but the split is by delivery destination, not by a rule that says "one FxOrder per merchant in the ConsumerReturn."

Consequence
Dispatch logic groups ItemReturns by deliveryDestination when creating FxOrders. A ConsumerReturn with 3 merchants creates 3 FxOrders (one per delivery destination), not necessarily 3 Dashers — the same Dasher can be assigned all three if routing permits.
4
QR Per ConsumerReturn, Not Per FxOrder
One QR code, backend resolves to correct Dasher's task

The consumer holds one QR code. The backend resolution logic uses the tuple (DasherID + QR token) to identify which FxOrder this Dasher should be handling. Multiple Dashers can each scan the same QR at different times, with each scan triggering only their FxOrder's liability shift.

This was an explicit choice over the alternative (one QR per FxOrder, consumer holds multiple QRs). The consumer-facing simplicity is significant — "show the Dasher your QR code" is a much simpler instruction than "show Dasher A their QR and Dasher B their different QR."

The resolution happens server-side: resolve(dasherId, qrToken) → FxOrder. If the Dasher is not assigned to any FxOrder for this ConsumerReturn, the scan is rejected.

Consequence
The resolve endpoint must be idempotent — multiple scans of the same (dasherId, qrToken) from the same Dasher return the same FxOrder without re-transitioning state. State transition is guarded: only from ASSIGNED → PICKED_UP, and only once.
5
historicalValue Immutability
Refund amount is snapshotted at ItemReturn creation from original CxOrder

When an ItemReturn is created, it snapshots historicalValue from the original CxOrder.lineItem.price. This value is immutable — it never changes after creation and is the sole basis for the refund amount.

This prevents two classes of manipulation: (1) price changes on the CxOrder (e.g., if prices were ever correctable post-purchase) cannot affect pending return amounts; (2) fraudulent attempts to re-submit the return with an inflated amount are blocked at the service layer.

The C&R service reads ItemReturn.historicalValue directly when firing the refund — it does not re-query CxOrder at settlement time.

Consequence
historicalValue is set exactly once in the ItemReturn lifecycle (at creation). All downstream services treat it as a read-only constant. Any refund reconciliation discrepancy should trace back to this field, making audits straightforward.
6
Polymorphic Order: CxOrder and ReturnOrder as Siblings
orderType: 'purchase' | 'return' on a shared Order abstraction

Rather than treating returns as a separate domain with its own order infrastructure, ReturnOrder is modeled as an Order with orderType: 'return' — a sibling to CxOrder (orderType: 'purchase').

This means: audit trails unify across order types, support tooling built for orders works on returns, analytics pipelines see both directions, and fraud systems can correlate purchase and return behavior on the same entity graph.

The operational fields that differ between purchase and return orders (FxOrders, delivery direction, etc.) live on the subtype — the shared Order base carries only the common fields (id, orderType, consumerId, timestamps, status).

Consequence
Order lookup endpoints need to filter by orderType to avoid polluting purchase history with returns. Consumer-facing order history shows only 'purchase' orders by default; 'return' orders surface through the dedicated Return History surface (accessible from the same app).
7
Eligibility as a First-Class Fraud Signal
Consumer behavior is a direct input to EligibilityCheck, not a post-hoc filter

The EligibilityCheck entity combines three signals: merchant return policy, item-level policy (e.g., perishables, opened electronics), and consumer behavior. Consumer behavior — return rate, monthly return value, and flagged items — is a first-class input, not an afterthought.

This means a consumer who has returned 90% of their orders in the past 30 days will see REQUIRES_REVIEW on eligible items, even if the merchant's policy is permissive. The signal gates them into a review queue rather than a hard block — reducing false positives while still allowing human review of edge cases.

The consumer behavior thresholds are configurable (not hardcoded) and owned by the Trust & Safety team — not the Returns team.

Consequence
Consumer Behavior Service must be queried at eligibility check time (pre-cart), not at commit time. This prevents a fraudulent consumer from discovering ineligibility only after reaching the QR code screen, which would be a poor UX and a potential trust issue.
8
Hub Inspection as Roadmap Feature, Not MVP Scope
deliveryDestination field ships in MVP schema with 'hub' enum value reserved

The MVP routes all returns directly to merchants (deliveryDestination: 'merchant'). However, the hub inspection model — where items go to a platform-operated centralized facility — is a natural evolution that materially changes unit economics and inspection quality.

The architectural decision was to bake the deliveryDestination enum into the ReturnFxOrder schema today, with 'hub' as a reserved value. When Phase 2 ships, dispatch routing changes to route eligible items to hub addresses. No entity schema migrations required.

The downstream systems (InspectionResult, MoneyMovementEvent, ItemReturn state machine) are hub-agnostic — they don't distinguish between a merchant-side or hub-side inspection. The inspectedBy field on InspectionResult accepts either a merchantId or a hubId.

Consequence
Dispatch service must be extended (not refactored) when hub launches: new routing logic branch for 'hub' destinations. All other services are already compatible. The team estimates a 1-sprint change to dispatch + hub onboarding work when Phase 2 is prioritized.
Architecture Principles

Guiding Principles for RLP

These principles informed every entity, state machine, and API boundary decision in this proposal.

⚡
Consumer State Is Sacred
ConsumerReturn is the consumer's record of the transaction. Operational re-routing, Dasher re-assignment, or fulfillment changes must never silently mutate consumer-visible state. All consumer state transitions are intentional and auditable.
🔒
Money Moves Only Through MoneyMovementEvent
No direct payment mutations. Every charge, refund, credit, or reversal goes through a MoneyMovementEvent with a type, state machine, and audit trail. The C&R service is the only writer; all other services read state.
📸
Immutable Financial Snapshots
historicalValue is snapshotted at creation. ReturnQuote is locked at commit. ServiceFeeBreakdown is written once. Financial history is never retroactively modified — only appended to via new events.
🌿
Extend, Don't Rewrite
RLP is designed as an additive layer on existing platform infrastructure. Polymorphic Order, shared Dasher dispatch, and existing payment rails are extended — not replaced. New entities are introduced; existing entities are read-only inputs.
🛡
Fraud Is a First-Class Concern
Eligibility, historicalValue immutability, commit-time fee capture, and consumer behavior signals are all fraud mitigations baked into the architecture — not added as patches. Returns are an inherently higher-risk surface than purchases.
🗓
Roadmap Hooks Are Schema Commitments
Known future features (hub delivery, exchange items) have explicit field reservations in the MVP schema. This prevents the "we'd need a migration" response when Phase 2 is prioritized. Schema is cheap to extend now; cheap to change later.

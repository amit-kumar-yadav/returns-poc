Money Movement Flow

Triggered at IN_TRANSIT — liability shifts at Dasher pickup scan

Card Refund

→ C&R fires against original CxOrder → refunds original payment method
⏱ 3–5 business days after Dasher pickup scan
Store Credit

→ DoorDash Credits issued (+10% bonus) — independent from CxOrder settlement
⏱ Instant upon Dasher pickup scan
Exchange
ItemReturn.refundMethod = exchange
→ ExchangeFxOrder created → forward logistics delivers replacement item
⏱ New item ships within 24h of pickup confirmation
Service Fee

→ Settled against ConsumerReturn — NOT deducted per ItemReturn
⏱ Shown in ReturnQuote before confirmation

ItemReturn Lifecycle States

PENDING_PICKUP → IN_TRANSIT → INSPECTED → REFUNDED → COMPLETED
PENDING_PICKUP → IN_TRANSIT = Liability Shift Event
Dasher scans item barcode on pickup. C&R triggered per ItemReturn independently. Each ItemReturn settles against its source CxOrder. Partial pickups are supported — one Dasher's FxOrder can be PICKED_UP while another is still PENDING.

CANCELLED is also a valid terminal state (policy rejection, customer cancellation, etc.)

Key Design Principles — RLP Vision

ConsumerReturn ≠ ReturnOrder
CR = consumer intent + QR + session state. RO = operational execution + Dasher assignment + financial settlement. The split allows re-routing, partial cancellation, and multi-Dasher flows without touching consumer-facing state.
ReturnFxOrder = 1 Dasher task (not 1 merchant)
Assignment engine splits by routing capacity. One Dasher can carry items from multiple merchants. One merchant's items can be split across Dashers. merchantIds is informational, not the partitioning key.
ItemReturn = unit of money movement
One C&R action per ItemReturn, reconciled against its original CxOrder. Service fee settles against ConsumerReturn — not per item. Each item refunds on its own timeline based on refundMethod.
CxOrder is immutable
Never modified after delivery. ItemReturn records carry all mutable state. returnState on the order history UI is a derived projection — not stored on CxOrder. This keeps the audit trail clean.
Fulfillment as abstraction (Task Framework)
ReturnFxOrder is a Task Framework task — the inverse of a forward delivery. Business logic defines what needs to happen; Task Framework handles Dasher assignment, routing, and state transitions.
